# Epistemic — 需求文档（PRD）

> 面向自己研究组的论文证据关系图与集体研究记忆工具
> v0.2 · 2026-07-15 · 随讨论滚动更新；本文档只描述当前设计

---

## 1. 定位

### 1.1 项目性质

- 自己研究组（Machine Learning 方向）的内部工具，不做商业化
- 建设团队 3 人，全员 AI 辅助开发；首批用户即本组（dogfood）
- 部署在一台公网服务器上

### 1.2 要解决的问题

1. **论文之间"到底什么关系"只存在于个人脑中。** 引用图只说"B 引了 A"，说不出"B 换掉了 A 的损失函数""B 发现 A 的结论只在小数据上成立"。人走了，这层理解就没了。
2. **谁读过什么、读到什么深度不可见**，导致重复阅读或互相以为对方读过。
3. **判断无法沉淀。** "这个结论可信吗？复现过吗？"的答案散落在群聊和个人脑中，三个月后从头再来。
4. **新成员没有可靠的入门阅读路径。**

### 1.3 一句话定位

给本组用的、每条边都带原文证据的论文关系图，加上组内阅读状态与判断的集体记忆。

### 1.4 不做（Non-goals）

- 不替代 Zotero 等文献管理器的收藏管理职能
- 不做公共平台、不建全球论文库
- 不给整篇论文打"对 / 错"标签——判断只落在 Claim、实验、方法组件级
- 无原文证据的 AI 推断永远不能成为"已确认事实"
- MVP 不做：idea 生成、阅读路径自动规划、投稿管理（未决，见 §11）、通知系统、移动端

## 2. 用户与场景

### 2.1 用户与权限

组内约 3—15 人。角色两级：**admin** 与 **member**。信息可见性两档：**私人**（仅自己）与**团队**（全组）。项目隔离用"项目标签 + 过滤"实现，不做硬隔离。

### 2.2 核心用户故事（MVP 必须全部支撑）

| # | 故事 |
|---|------|
| US1 | 我把一个 arXiv 链接丢进去，半分钟内它有了元数据、版本家族、参考文献，后台开始抽取 DNA。 |
| US2 | 我打开一篇论文，能看到它改进了谁、被谁反驳、方法来自哪，每条关系点开能跳到 PDF 原文证据位置。 |
| US3 | AI 提的候选关系，我在审核队列里一键接受 / 改类型 / 拒绝，全程键盘操作。 |
| US4 | 我对某条 Claim 标"复现失败"并附实验链接，其他人能看到、能回复、能反对。 |
| US5 | 组会前我扫一眼：这周新论文谁读了、读多深、留了什么批注。 |
| US6 | 新同学打开项目页，按确认过的 prerequisite 关系倒排出一份前置阅读清单（MVP 允许半手动整理）。 |

## 3. 产品原则

1. **三层信息严格分级**：公开事实 / 团队记录 / AI 候选——数据层用字段区分，视觉层用样式区分，永不混淆。
2. **无证据不入图**：每条 AI 抽取的字段和关系必须绑定原文 span（页码 + 文字），点击可跳转 PDF 对应位置；给不出证据的输出直接丢弃。
3. **判断落在 Claim / 方法组件级**，不给整篇论文打分。
4. **零确认也有用**：纯自动层（元数据 + 引用 + DNA 抽取）自身就要成立；人工确认是增益，不是前提。
5. **两种图视图各司其职**：全局语义地图负责概览探索（位置 = 相似度，LOD 分级渲染），Ego 聚焦视图负责关系审读；**画出来的边只有断言关系，相似度只决定位置、永不画成边**。
6. **所有编辑可追溯、可撤销、可争议**。
7. **版本家族默认折叠**为一个节点，展开才见 arXiv v1/v2/camera-ready。
8. **录入靠 AI 提议 + 人一键裁决**，不靠人手填表单。

## 4. 数据模型

### 4.1 核心实体（10 个图实体 + 2 张状态表）

| 实体 | 说明 |
|------|------|
| ResearchWork | 论文家族（一项工作的抽象），图上的默认节点，**团队认知容器**：批注、判断、阅读状态都挂在它上面、随时可取 |
| PaperVersion | 具体版本：arXiv v1/v2、会议版、期刊扩展；属于一个 Work |
| Person | 组内成员（兼作者消歧锚点） |
| Project | 组内项目 / 方向标签 |
| Claim | 可独立讨论的主张，属于某个 Work |
| Method | 方法 / 方法组件（parent 自引用表达组件层级） |
| Dataset | 数据集 |
| RelationAssertion | 关系本身（具体化为节点，见 §4.3） |
| EvidenceSpan | 原文证据：version_id + 页码 + 文本 + 坐标 |
| Annotation | 批注，带类型（笔记 / 猜想 / 问题）与可见性（私人 / 团队） |

状态表（偏协作、不上图）：

- **ReadingStatus**：member × work 的阅读状态（§5.5）
- **Review**：成员对关系或 Claim 判断的确认 / 反对；同一对象上出现相反 Review 即为"争议"，不单设争议实体

暂不建的实体：Venue（论文上用普通字段）、Experiment / Result / Metric（并入 Claim 的结构化描述）、Submission（未决，见 §11）、Idea；复现尝试用阅读状态"复现过"+ 批注表达。

### 4.2 三层来源分级

所有实体和关系带：

```
source        ∈ { public_fact, team_record, ai_candidate }
review_status ∈ { unreviewed, confirmed, disputed, rejected }
created_by      （人 或 模型+版本号）
```

铁律：`ai_candidate` 不经人工确认永不升级为 `confirmed`；投稿 / 拒稿类信息禁止 AI 推测，只能来自公开元数据或人工录入。

### 4.3 关系类型（白名单 12 种）

收录标准：每种类型必须有独立的查询或展示行为；细分差异（如"改进精度"vs"改进效率"）进 `aspect` 字段而不是新类型。AI 不允许发明白名单之外的类型。

| 组 | 类型 | 说明 |
|----|------|------|
| 结构 | cites | 来自元数据源，自动导入，不需人审，图上默认隐藏 |
| 版本 | version_of | 家族归属，自动判定 + 人工可改 |
| 方法 | uses_method_from | B 使用了 A 的方法 / 组件 |
| 方法 | improves_on | 带 `aspect: accuracy \| efficiency \| generality \| simplicity` |
| 方法 | alternative_to | 平行竞争方法，无继承关系 |
| 实验 | uses_dataset_from | |
| 实验 | compares_against | 作为 baseline 比较 |
| 实验 | reproduces | 复现（成功） |
| 实验 | fails_to_reproduce | 高风险类型，见 §6.4 |
| 论证 | supports_claim | 指向具体 Claim，不指向整篇论文 |
| 论证 | contradicts_claim | 高风险类型 |
| 阅读 | prerequisite_for | 前置阅读关系，主要靠人工 + AI 建议 |

RelationAssertion 的字段：

```
type, aspect, scope（作用范围，如 "attention complexity"）, explanation,
members[{entity_id, role}]      ← 天然支持多元关系
evidence[], confidence, source, review_status,
created_by, model_version, created_at
```

**锚点与投影**：关系存储在它实际成立的最细粒度（如"B 反驳 A 的 Claim#2"就锚到 Claim#2）；每个成员记录其所属论文（anchor work），因此任何关系都能自动投影为 Work 层的边供图显示。粗粒度关系可事后细化锚点，图的形态不变。

**超边**：members 数组支持"A 的架构 + B 的目标函数 → D"这类组合关系；当前 UI 和抽取只做二元关系，开放组合关系时数据零迁移。

### 4.4 去重与版本家族

标识符优先级：DOI > arXiv ID > 标题 + 作者 + 年份模糊匹配 > PDF 指纹。
家族判定信号：标题相似 + 作者重合 + arXiv journal-ref / DOI 关联。
合并必须**可逆**：保留合并历史，支持拆分、指定主版本。

## 5. 核心体验

### 5.1 导入

两个入口：

1. **清单批量导入**：粘贴"标题 + 网址"的论文清单 → 解析 → 预览确认 → 入库（清单实际格式未决，见 §11）
2. **单篇快速添加**：arXiv URL / ID 或 DOI

URL 解析策略：arXiv 链接自动补全元数据并拉取 PDF；其他链接（OpenReview、ACL Anthology、出版社页）尽力解析元数据，PDF 支持手动上传补充。

管线：标识符解析 → 元数据补全（Semantic Scholar 为主、arXiv API 为辅）→ 去重 → 家族识别 → 参考文献获取 → GROBID 解析 PDF → DNA 抽取 → 候选关系生成 → 距离维度更新。全部异步，导入即刻可见条目，AI 结果就绪后增量出现。

### 5.2 论文卡片（Paper DNA）

四块：

- **基本**：标题、作者、年份、会议 / 期刊（字段级）、版本家族、代码 / 数据链接
- **研究**（每字段带证据跳转）：研究问题、主要贡献、方法与组件、数据集、主要结论（即 Claims）、局限
- **团队**：谁读过 + 深度、判断、思考流（批注：笔记 / 猜想 / 问题）、所属项目
- **图谱**：前置、改进、被改进、复现、冲突（各为一行可点击的关系列表）

### 5.3 图视图：全局语义地图 + Ego 聚焦视图

两种视图共用距离引擎（§5.6）与同一套视觉语言。**画出来的边只有断言关系（有证据、可审核）；相似度只决定位置。** 位置说"它们住同一街区"，边说"我们确切知道它们什么关系"。

#### 5.3.1 全局语义地图（概览 / 探索）

- 团队库全部论文一屏铺开，**位置 = 距离引擎加权输出**（力导向，弹簧长度 = 距离）
- 布局稳定性：固定随机种子 + 增量布局——老节点基本不动，新论文"落"到邻居附近，保住空间记忆
- 缩放分级渲染（LOD）：远景只有圆点 → 中景浮现标题 → 近景才画断言边与角标
- 点节点 → 侧栏卡片（DNA + 团队思考流）；双击 → 以它为中心进入 Ego 视图
- 主维度下无邻居的论文停靠在地图边缘"**未接入区**"（按导入时间排列）——组的知识结构尚未消化它们，本身是信息；可开启"主题引力"给它们近似位置（§5.6）
- 团队 overlay（可开关）：节点圆点数 = 组内已读人数，灰 = 无人碰过，红角标 = 有未决争议

#### 5.3.2 Ego 聚焦视图（关系审读）

- **中心可以是任何实体**：论文（默认）、Claim、Method、Dataset——"下钻"= 换焦点，配面包屑导航
- 经典力导向，弹簧长度接距离引擎；默认一跳，可扩两跳
- **可见节点上限 30，由算法选人**：relevance 分 = 已确认 > AI 候选；语义关系 > cites；× 视图模式权重（找前置 / 找争议 / 找演化）；+ 团队信号（读过 / 星标 / 有批注加分）
- 溢出节点按 **(关系类型 × 方向)** 聚成语义组节点（"改进它的 +8 篇"），点击展开该组
- **边 = 断言束（bundle）**：同一对论文间同语义组的断言合并为一条视觉边，带数量角标，点击展开清单逐条看证据；最多按语义组并行 3 条边
- 样式：AI 候选 = 灰虚线；单人确认 = 浅实线；多人确认 = 深实线；争议 = 红实线；cites 默认隐藏
- 版本家族折叠为单节点（复合节点），双击展开
- 点击边 → 侧栏证据面板：解释 + 原文引文 + 跳 PDF + 确认 / 反对按钮
- prerequisite 阅读图是**独立派生视图**（列表为主 + 小图为辅）：写入时环检测，读序拓扑排序，不混入 ego 画布

### 5.4 审核队列（一等公民界面）

人机协作的主战场，独立入口：

- 按论文或按批次进入队列；每条候选展示：类型 + 方向 + 一句话解释 + 原文证据
- 操作：接受 / 改类型 / 调转方向 / 拒绝（理由可选）——全键盘（j/k 移动，a 接受，r 拒绝…）
- 每次裁决记录 model_version，为抽取质量评估积累数据

### 5.5 阅读状态与批注

阅读状态 5 档 + 1 标记：**未读 / 略读 / 精读 / 复现过 / 需复核**，外加"星标"flag。

批注：PDF 高亮 → 批注（私人 / 团队），带类型：**笔记 / 猜想 / 问题**；高亮可一键"升格"为 Claim 或某关系的证据。批注、判断、阅读状态都挂在论文节点上，随时可取。

Claim 判断枚举 6 种：`supported / partially_supported / contradicted / not_reproduced / concern（附说明）/ unclear`。每个判断绑定：人、日期、证据或实验链接、适用条件。

### 5.6 距离引擎

驱动两种图视图的节点位置。

**主维度（两个，加权混合）：**

| 维度 | 计算 | 何时可用 |
|---|---|---|
| 引用耦合 | 共同参考文献 + 共同被引的归一化重合度（bibliographic coupling / co-citation） | 导入当天（引文元数据自带） |
| 方法谱系距离 | 沿方法组关系（uses_method_from / improves_on / alternative_to）的图距离；已确认边长 1、未审候选边长 2，≤ 4 跳，分数 = 1/d | 随 AI 候选与人工确认逐步变密 |

**孤岛与主题引力：**主维度下无任何邻居的论文默认停靠"未接入区"。另有**主题引力开关（默认关闭）**：开启后，仅对无邻居论文施加基于文本 embedding 的弱引力使其获得近似位置；主维度有信号的论文不受影响。embedding 复用候选召回（§6.3）同一套向量，无额外成本。

**实现约定：**

- 每维度**预计算**每篇论文 top-32 邻居 + 分数；导入、关系状态变化时增量更新
- 前端拿到各维邻居表后，**调权重滑杆是纯前端重排**，不回服务器
- 权重组合可保存为命名"视角"（如"谱系视角"），全组共享
- 诚实性：2D 是近似，悬停显示各维真实分数；位置是导航提示，真相在边和证据里

## 6. AI 管线

### 6.1 解析

GROBID（docker）→ TEI XML → 章节化入库，保留页码与坐标映射（证据跳转依赖）。表格 / 公式抽取质量有限：保留原 PDF 供人工复核，不追求完美解析。

### 6.2 Paper DNA 抽取

Chat Completions API + 强约束 JSON schema（`response_format.json_schema`）。每个字段必须携带 `source_text + page`；给不出证据的字段直接丢弃（原则 2）。每篇只抽一次并缓存，模型与 prompt 版本入库。

### 6.3 候选关系：两条路，引文上下文优先

1. **引文上下文分类（主力）**：对每处引用，取其所在句段 → 分类成白名单关系类型。便宜、准确率高、天然自带证据 span，覆盖所有直接引用关系。
2. **向量召回 + 成对判定（限额）**：标题 + 摘要 + 方法字段做 embedding，召回 top-K 非引用候选（K≈10）→ LLM 成对深度比较。覆盖"相似但互不引用"的关系。

不做全库两两比较（1000 篇 ≈ 50 万对）。

### 6.4 阈值与高风险类型

- confidence < 0.5：丢弃；0.5—0.75：默认隐藏，可在"更多候选"里展开；> 0.75：进审核队列
- **高风险类型**（fails_to_reproduce / contradicts_claim）：无论置信度，一律需人工审核，确认前不显示在图上

### 6.5 评估

动工前先建**小测集**：组内最熟的 20—30 篇论文，人工标注 DNA 字段 + 关系对。先测再扩，不达标就收缩关系类型而不是灌低质量边。指标：字段准确率、证据定位准确率、关系 precision / recall、人工接受率。

## 7. MVP 范围

### 7.1 验收演示（10 步）

1. 建组、加 3 名成员
2. 导入 100 篇论文（清单批量粘贴 + arXiv 单篇）
3. 自动去重 + 版本家族识别正确率抽查
4. 打开论文卡片：DNA 字段齐全，点字段跳到 PDF 原文位置
5. 打开全局语义地图：缩放平移、LOD 三级渲染正常；双击一篇论文进入 Ego 视图，一跳 / 两跳展开
6. 按关系类型过滤，隐藏 / 显示 AI 候选
7. 审核队列处理 20 条候选（接受 / 改 / 拒）
8. 两人对同一条关系意见相左 → 该边变红显示争议
9. 设阅读状态、写团队批注（含"猜想"类型）、回复
10. 项目页查看团队覆盖：谁读了什么、哪些无人读

### 7.2 暂不做

投稿管理（未决）、自动化爬虫与持续追踪、BibTeX / Zotero 导入、阅读路径自动生成、idea 生成、超边组合结点 UI（数据模型已支持）、地图聚类色晕（LOD 只做点 / 标题 / 边三级）、Neo4j、OpenReview 接入、通知系统、移动端、公共分享。

### 7.3 后续阶段（顺序示意，非承诺）

- P2：自动化爬虫与持续追踪（新论文自动入库分析）、OpenReview 接入、距离引擎新维度（数据集重合等）、地图聚类色晕、超边组合结点 UI、争议工作流完善、BibTeX / Zotero 导入
- P3：前置阅读路径生成、团队知识覆盖分析
- P4：基于已确认图谱的 idea 辅助

## 8. 技术选型

| 层 | 选型 | 说明 |
|----|------|------|
| 前端 | Vite + React + TypeScript + Cytoscape.js + PDF.js | SPA，内部工具不需要 SSR |
| 后端 | Rust：axum + sqlx + tokio | 编译期校验 SQL；强类型是 AI 生成代码的天然护栏 |
| 数据 | 单 PostgreSQL（+ pgvector） | 关系数据、图（邻接表 + 递归 CTE）、向量、任务队列全在一库 |
| 任务 | jobs 表 + `FOR UPDATE SKIP LOCKED`，worker 独立二进制 | 不引入 Redis / 消息队列 |
| 存储 | 服务器磁盘卷（PDF），定期备份 | 不引入 MinIO |
| 解析 | GROBID（docker，HTTP 调用） | |
| LLM | OpenAI Chat Completions 兼容 API（自封装 HTTP 客户端） | 型号与用法见 docs/DEV.md |
| 部署 | docker-compose + Caddy 自动 HTTPS | 公网服务器，账号邀请制 |

选 Rust 的实际收益是类型安全 + sqlx 编译期 SQL 校验 + 单二进制部署——对"三人全员 AI 生成代码"的开发方式，编译器是最可靠的第一道审查（系统瓶颈在外部调用，吞吐不是理由）。

**明确不用**：Neo4j、微服务、K8s、Redis、独立向量库、独立搜索引擎（Postgres FTS 够用）。

规模假设：1 个组、≤ 20 用户、≤ 5,000 篇论文、≤ 10 万条关系。单 Postgres 毫无压力；两跳邻域递归 CTE 毫秒级。若未来需要图算法（社区发现、中心性），加只读投影，Postgres 始终是唯一事实源。

| 风险 | 应对 |

## 10. 里程碑与 Go / No-Go

| 阶段 | 内容 | 完成标志 |
|------|------|----------|
| M0 定案 | 本文档定稿、数据模型 SQL 草稿、20—30 篇测试论文标注 | 全组同意范围 |
| M1 骨架 | 导入 + 元数据 + 去重 + 家族 + 论文列表 + 阅读状态 | 无 AI 也可日常使用 |
| M2 AI 层 | GROBID + DNA 抽取 + 证据跳转 | 抽取质量过小测集门槛 |
| M3 图与关系 | 引文上下文候选 + 审核队列 + 全局地图 + Ego 视图 + 距离引擎 | 跑通 §7.1 十步演示 |
| M4 Dogfood | 全组导入真实文献库，运行 4 周，每周复盘 | 见下 |

**Go / No-Go（M4 后评估）**：

1. 组里 ≥ 3 人每周主动使用（非被要求）
2. AI 候选人工接受率 ≥ 60%
3. 组内共识：优于现状（Zotero + 群聊 + Notion）
4. 若大家只用搜索和摘要、不碰关系审核和阅读状态 → 差异化不成立，停止扩展图谱功能，反思定位
